# Infrastructure-oriented Task Management System

### Описание проекта
Данный проект представляет собой веб-приложение для управления задачами, спроектированное с упором на микросервисную архитектуру, контейнеризацию и построение системы глубокого мониторинга (Observability). Основной целью разработки являлось создание отказоустойчивой среды с разделением логики обработки запросов, хранения данных и сбора телеметрии.

### Цели разработки
1. Реализация REST API на базе Flask для управления жизненным циклом задач.
2. Организация сетевого взаимодействия между изолированными контейнерами внутри Docker-сети.
3. Развертывание централизованной системы сбора логов и метрик для обеспечения прозрачности работы инфраструктуры.
4. Настройка реверс-проксирования для оптимизации обработки входящего трафика и безопасности.

### Технологический стек
* **Backend:** Python 3.11, Flask, SQLAlchemy.
* **Database:** PostgreSQL.
* **Frontend:** JavaScript (Fetch API), HTML5, CSS3.
* **Reverse Proxy:** Nginx.
* **Containerization:** Docker, Docker Compose.
* **Logging:** Grafana Loki, Promtail.
* **Metrics:** Prometheus, cAdvisor.
* **Visualization:** Grafana.

### Архитектурные особенности и внедренные решения
* **Разделение трафика:** Nginx настроен как единая точка входа. Статический контент отдается напрямую, запросы к API проксируются на бэкенд-сервер по внутренним адресам Docker-сети.
* **Управление данными:** Использование SQLAlchemy в связке с Flask-Migrate (Alembic) позволяет управлять схемой данных через механизм миграций, исключая ручное редактирование таблиц и обеспечивая целостность БД.
* **Централизованное логирование (PLG Stack):** Настроен автоматический сбор stdout/stderr потоков всех Docker-контейнеров. Данные агрегируются в Loki и доступны для анализа через Grafana с использованием языка запросов LogQL.
* **Мониторинг ресурсов:** Интеграция cAdvisor обеспечивает сбор метрик потребления CPU, RAM и сетевой активности на уровне каждого отдельного контейнера.
* **Безопасность инфраструктуры:** При монтировании системных директорий хоста в контейнеры мониторинга строго соблюдается режим Read-Only (:ro). Доступ к API ограничен через конфигурацию прокси-сервера.

### Реализованные сценарии мониторинга
1. Мониторинг HTTP-статусов: отслеживание частоты возникновения ошибок 4xx и 5xx в реальном времени.
2. Анализ производительности бэкенда: расчет RPS (Requests Per Second) через PromQL.
3. Контроль утилизации ресурсов: отслеживание лимитов памяти контейнеров для предотвращения OOM-событий.
4. Траблшутинг через логи: быстрая фильтрация сообщений об ошибках по именам конкретных сервисов.

### Трудности, возникшие в процессе реализации
* **Маршрутизация Nginx:** Столкнулся с проблемой некорректной обработки URI при использовании завершающего слеша (trailing slash) в директиве proxy_pass. Проблема была решена путем нормализации путей в конфигурации Nginx, что обеспечило корректное сопоставление роутов Flask.
* **Права доступа в Docker:** Процесс Promtail изначально не имел доступа к чтению логов контейнеров из-за ограничений прав доступа в ОС хоста. Проблема устранена путем настройки запуска сервиса от имени пользователя root внутри контейнера.
* **Синтаксис конфигурационных файлов:** Возникали ошибки парсинга конфигурации Promtail из-за чувствительности формата YAML к отступам при вставке через терминал. Проблема решена автоматизацией записи конфигов через HEREDOC.
* **Инициализация базы данных:** При первом запуске возникла ошибка отсутствия таблиц (relation "tasks" does not exist). Проблема решена внедрением этапа миграций в жизненный цикл развертывания через flask db upgrade.

### Инструкция по развертыванию
1. Склонировать репозиторий.
2. Создать файл `.env` на основе примера и настроить параметры доступа к БД.
3. Запустить проект: `docker compose up -d --build`.
4. Выполнить миграции базы данных: `docker compose exec backend flask db upgrade`.
5. Доступ к приложению: `http://localhost`.
6. Доступ к панели мониторинга (Grafana): `http://localhost:3000`.
